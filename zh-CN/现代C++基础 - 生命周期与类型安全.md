# 存储期

- 存储期表示对象所使用的存储将会存在多久。
- 存储期有四种：
- 静态存储期：全局变量、函数/类中的静态变量。会在进入 `int main` 之前构造，并在退出 `main` 之后析构。异常中止不会调用析构函数。
- 自动存储期：属于块作用域的变量或函数参数。在定义时构造，并在当前作用域退出时析构（例如函数返回）。
- 动态存储期：可通过 `new` 或其他分配函数创建对象。你只能持有相应指针，销毁指针并不会自动释放动态内存。
- 线程存储期：在线程创建时构造，在线程退出时析构；以后会讲到。

# 生命周期

- 对象的生命周期在对齐与大小合适的存储被分配且对象完成初始化时开始。
- 包括可以默认初始化的类型（例如 `int`、带有默认初始化器的类）。
- 对象的生命周期在对象被销毁、或析构函数被调用、或其存储被释放或被非嵌套对象复用时结束。
- 若 `a` 是 `b` 的成员，则 `a` 嵌套在 `b` 之内。
- 特别地，尽管引用要到离开作用域才结束其“引用自身的生命周期”，但若底层对象已结束生命周期仍去使用它，则属于悬空引用。
- 这是因为访问引用与访问底层对象是同一回事，而在生命周期之外访问属于未定义行为（UB）。更精确地说，某些操作仍被允许，例如把引用绑定到另一引用、访问静态成员，但利用它们并不妥当。

# 生命周期

- 如我们所见，临时对象只存活到语句结束（即到 `;` 为止）。
- 这就是为什么把 `const&` 或 `std::function_ref` 用作参数是安全的，即便传入的函数对象是临时的。
- 此外，返回的临时对象的生命周期可以被某些引用延长，例如我们学过的 `const&`。
- 再次强调：这要求“返回的是临时对象”；返回指向局部变量的引用或指针仍然是错的。

- 边角情况：在调用构造/析构函数之后、在该函数返回之前，对象的生命周期已经结束，而这是唯一仍可使用其成员（在若干限制下）的地方。
- 例如我们已知：不能在构造/析构函数里调用虚函数。

# 生命周期

- 边角情况：
- 情况 1：若你构造的对象与原先对象类型相同（忽略 cv 限定），且它们占用完全相同的存储，则原来的名字、指针与引用仍然有效（这与使用赋值运算符非常相似……）。

# 严格别名规则

- 基于生命周期，我们可以对指针做优化。
- 例如，`int* a` 与 `float* b` 不可能引用同一对象，因此编译器可以假定它们不同。
- 于是 `*a += *b; *a += *b;` 可优化为 `*a += 2 * *b;`，而无需担心它们别名导致最终变成 `*a += 3 * *b`。
- 这称为严格别名规则：即若指针不构成别名、或不是以成员方式相互包含，则编译器可以自由假定它们指向不同对象。
- 例如，指向类与其成员类型的指针也不会被这样优化，例如 `class A{ int a; float b; };` 下的 `A*` 与 `float*`。
- 即便当前未必如此，编译器未来也可能优化掉，例如 stackoverflow 上的问题，Clang 14 就会这么做。

你可以使用 `-fno-strict-aliasing` 来关闭它。

你可以在 C 里使用 `restrict` 关键字（奇怪的是 C++ 标准里没有，但各编译器都支持）表明指针与其他东西不重叠，例如 `uint8_t* restrict`

# 注意 view 的生命周期

- 有时你可能在函数里返回由范围适配器生成的 view（例如 `v | stdv::reverse`）。
- 那么 view 的生命周期是什么？
- 对左值，它与该左值本身相同，即此处的 `v`。
- 因此若你创建局部变量并返回指向它的 view，则无效。
- 对右值，它类似于返回值，因此总是安全。
- 例如 `std::vector{1,2,3} | stdv::reverse`；`std::move(v) | stdv::reverse`。
- 以后会讲 `std::move` 是什么！
- 本质是 `stdv::all`：所有范围适配器都会首先尝试用 `stdv::all` 把范围转换为 view。
- 对 view 本身什么也不做；对左值范围会创建 `stdr::ref_view`；对右值范围会创建 `stdr::owning_view`。
- 从名字就能看出生命周期问题！`ref_view` 等价于引用，而 `owning_view` 持有值本身。

# 多重继承

- 多重继承自诞生起就充满争议。
- C++ 提供了强大的多重继承语法，考虑了大多数可能情形，并允许你实现任何想要的设计。
- 然而自由伴随代价；若随意使用多重继承，通常会很复杂，损害可维护性，有时还会引发令人震惊的问题。
- 在许多其他 OOP 语言如 C#、Java 中只允许单继承，同时用 interface 提供多重继承的一种特例。
- 这称为混入模式（Mixin Pattern）；若要在 C++ 里用多重继承，这通常是最常见、也最推荐的方式。

# 隐式转换

- 一些隐式转换是自动的（即标准转换），另一些是用户定义的。
- 我们学过例如 `operator float()`。当然若声明为 `explicit`，隐式转换会被禁用。
- 不那么重要：粗略地说，隐式转换会先尝试标准转换；若表达式仍非法，再尝试用户定义转换；若仍非法，再尝试标准转换。通常若标准转换—用户定义转换—标准转换（s-u-s）序列合法，则表达式合法。
- 我们现在讲标准转换；共有四类：
- 1. 左值到右值转换、数组到指针转换、函数到指针转换。
- 2. 数值提升与转换。
- 3. （带异常说明的）函数指针转换。
- 4. 限定转换。
- 标准转换会按需一步步尝试以让表达式合法。

# 隐式转换

- 对第一类：
- 左值到右值转换：我们还没深入值类别，因为它与移动语义强相关，这里只粗略讲。
- 以前我们学到：左值是可以出现在赋值左侧的东西（忽略 `const`），右值不能（例如 `2 = a` 非法）。
- 这种转换表示一般左值可视为纯右值；直观上是因为左值可以出现在右侧。
- 这里不再深挖；只需记住一般左值可隐式视为纯右值以使表达式合法。
- 数组到指针转换：`T arr[N]` 可丢掉大小并隐式转为指针 `T*`。
- 这就是为什么你能做例如 `arr + 3` 这样的指针算术。
- 这也称为“数组到指针退化（decay）”。
- 注意只有第一维可退化，例如 `int a[3][4]` 只能隐式转为 `int (*)[4]`。

非引用模板形参也会退化，例如 `(T a, T b)` 永远不会自动推导出 `T = xx&`（尽管你可以用 `<T&>` 显式指定）。

# 隐式转换

换句话说，`auto` 的类型推导与类型形参的推导完全相同（除了 `std::initializer_list`，模板永远不会推导它）。
- 函数到指针转换：函数类型（非成员函数）可隐式转换为其指针。
- 这就是为什么例如 `FuncPtr a = func` 与 `FuncPtr a = &func` 都可以，但成员函数需要显式 `&`。
- 所以 decay 实际意味着：
- 数组/函数 -> 指针
- 或其他类型先去掉引用，再去掉 cv 限定。
- 可用 `std::decay_t<T>` 得到 decay 后的类型。
- `auto a = xx;` 也会让推导类型 decay，而 `auto&` 不会！
- 有时称为 decay-copy。
- 但结构化绑定（`auto [a, b]=xx`）“看起来”不衰减？当成员是 `const` 时，你得到的仍是 `const`。
- 记得吗？它等价于 `auto anonymous = xx;`，而 `a`、`b` 是其成员别名，因此 decay 只发生在 `anonymous` 上（对用户不可见）。

- 2. 关于数值类型的转换：
- 首先是提升（promotion）；提升优先级更高且不会损失精度。

# 隐式转换

# dynamic_cast 与 RTTI

- 我们看到 `static_cast` 沿继承链不检查合法性；即便生命周期里没有实际派生对象，也能 cast 到派生类。
- 它只用 UB 来约束，这太弱了！
- `dynamic_cast` 试图解决这一点；在不合适时转换会失败。
- 具体地，引用转换失败会抛 `std::bad_cast`，指针转换失败返回 `nullptr`。
- 这比 UB 更强，也更方便找 bug！
- 因此为在运行期做类型检查，会保留 RTTI（Run-Time Type Information/Identification）。
- 注意 `dynamic_cast` 只能用于多态类型，因为 RTTI 依赖虚指针一类机制。

# dynamic_cast

- 然而安全有代价。
- 使用 RTTI 的 `dynamic_cast` 至少比 `static_cast` 慢 10 倍，某些情形下甚至慢成百上千倍！
- 前者常见于沿继承链转换到真实底层类型（通常情况）；后者常见于继承图中的“侧向转换（sidecast）”（即多重继承）。
- 常常 `dynamic_cast` 意味着设计缺陷，即原类型无法仅靠多态表达所有行为；因此通常最好不要频繁使用它。
- 性能敏感项目可能自己写 downcast（例如 LLVM）。
- 1. 可在多态类型中做 downcast。
- 假设底层对象真实类型是 `T`，而我们持有 `B*`。
- 若把 `B*` 转为不派生自 `T` 的另一类型，则失败。
- 否则转换成功，可得到指针。
- 注意不应在构造/析构函数里用 `dynamic_cast`，因为此时 vptr 尚不完整。

# dynamic_cast

- 2. 也可在多态类型与多重继承中做侧向转换。
- 例如我们有一个 `ElephantSeal`，由 `Elephant&` 引用。
- 现在你需要把它当 `Seal&` 用；你不能用例如 `static_cast<Seal&>`，因为 `Seal` 与 `Elephant` 没有直接继承关系。
- 即在上转之后继承路径不完整；仅凭 `Elephant`/`Seal` 无法知道它是不是象海豹。
- 于是你必须 `static_cast<ElephantSeal&>`，再隐式转为 `Seal&`。
- 这很高效，但在复杂情形下你可能不知道确切类型，因而危险，此时就需要直接侧向转换。
- 这里可以直接 `dynamic_cast<Seal&>`。

- 内部过程仍是先转到底层类型（此例为 `ElephantSeal`），再上转。
- 当然仍要检查上转是否存在二义性。
- 若上转失败，整个转换失败。
- 其他小功能：
- 3. 可用 `dynamic_cast` 把指针转到最派生类（即指向底层对象的指针）。
- 通过 `dynamic_cast<void*>(…)`。
- 但这很少用，因为你仍需要确切类型再 `static_cast`，`void*` 本身做不了什么。
- 4. 也可从派生类转到基类，就像隐式转换。
- 这两种不需要 RTTI，因此与 `static_cast` 一样快（尽管有点多余……）。

# RTTI

- C++ 也提供通过 `<typeinfo>` 直接使用底层运行时类型信息的方式。
- 仍建议在受限场景使用，因为性能代价高。
- 可用 `operator typeid(xxx)` 得到 `const std::type_info`。
- 该运算符类似 `sizeof`。
- `std::type_info` 可比较相等（检查两类型是否相同）、调用 `.name()` 取名字、`.hash_code()` 取哈希、`.before()` 按强序比较（平台相关）。
- 但不能直接用于关联容器，因为它没有 `std::hash`/`operator<` 特化。
- 若真要用作关联容器的键，可用定义于 `<typeindex>` 的 `std::type_index`，它只是包装器。
- 其构造函数接受 `std::type_info`。

# RTTI

- 注意 `type_info` 对象是只读的；不能拷贝或构造它们。
- 即 RTTI 信息完全由编译器持有。
- 可用于调试，尤其通过 `.name()`。

- 最后，RTTI 对共享库（跨“模块边界”）不友好。
- 例如自 GCC 3.0 起，符号相等性按地址比较而非按名字。为在多个 `.obj` 中只保留一份符号，链接时会合并（如静态库）。
- 然而共享库常用运行期加载（例如 ICS 见过的 `dlopen`）。

# RTTI

- 为快速加载共享库，许多步骤被省略，其中包括解析不同 RTTI 符号。
- 例如若在 `dlopen` 时不加 `RTLD_GLOBAL`，默认是 `RTLD_LOCAL`。
- 于是符号不会与其他符号重定位，你会得到两个不同的 RTTI 符号地址！
- 于是当你在程序里构造对象，却把它传入从共享库加载的函数并在库内做 `dynamic_cast` 时，可能得到意外行为。
- 也就是说，即便你传入的 `Base*` 底层是 `Derived`，`dynamic_cast<Derived*>` 也可能返回 `nullptr`。因为 `dynamic_cast` 找到的是程序里的 RTTI 符号地址，与库里的 RTTI 符号比较会不相等（即转换失败）。
- 同理，在库里构造对象再在程序里持有并 `dynamic_cast` 也危险。

# RTTI

- `typeid` 比较也有类似问题；如何解决可见 GCC 文档。
- 最后：总之 RTTI 在运行期慢、在加载期也慢（跨模块边界使用时），许多项目不鼓励使用。
- 例如 Qt 用 `qobject_cast`。
- 记得 C++ 的哲学“只为使用的特性付费”吗？许多人认为 RTTI 违背这一点。
- 总之谨慎使用 RTTI；若你想用它，尤其写库或性能敏感路径时，先想想设计能否改进。

# const_cast

- 例如在我写的一个 FTP 程序里。
- 理论上不应修改用户命令，因此处理函数用 `std::string_view`，本质是 `const char*`。
- 然而只有一个例外：命令 `open`，即 `open IP port`。

# const_cast

- 第二种情形是用第三方库：作者参数忘了写 `const`，但事实上不写（文档明确说明或你能看源码）。
- 那就可用。

# reinterpret_cast

- `reinterpret_cast` 用于处理不同类型指针间的转换，因生命周期而危险。
- 1. 从对象指针转到另一类型，即 `reinterpret_cast<T*>(xxx)`。
- 这与 `static_cast<T*>(static_cast<(cv) void*>(xxx))` 相同，因此仍需要 `xxx` 与 `T*` pointer-interconvertible。
- 记得吗？若要把失去生命周期的旧指针转到新指针，可能还需要 `std::launder`。
- 2. 从一种函数指针转到另一种函数指针；或成员指针之间互转。
- 除语言允许隐式进行的函数指针转换外，只有往返转换才有效。即若要调用函数，必须再转回原始类型。

# reinterpret_cast

- 在许多平台（例如 POSIX）上，把它们转为 `void*` 再转回可得到原指针；但标准不强制，因此在某些古怪系统上往返可能编译失败。
- 也要注意可能的对齐问题。
- 3. 指针与整数互转。
- 若整数足够大，指针可用 `reinterpret_cast` 转为整数。
- 在 C++ `<cstdint>` 中，`std::uintptr_t` 定义为这样的整数类型。
- 该整数可再转回得到原指针。
- 若传入随机整数或宽度不足的整数，则 UB。
- 最后，从 `0`/`NULL` 做 `reinterpret_cast` 是 UB；请用 `nullptr`/隐式转换/`static_cast`。
- 4. 引用也可转换；与指针转换同理。

# reinterpret_cast

- 你会发现由于生命周期的限制，其功能被大幅收紧。
- 更宽松地说，别名情形也可以，如严格别名规则所学，例如可用 `reinterpret_cast<unsigned int&>` 引用 `int`。
- 你不能例如用 `reinterpret_cast<float&>` 去看二进制（像过去常做的那样）；需要 `std::bit_cast` 或 `std::memcpy`。
- 但这些规定都与 UB 相关，意味着它并不阻止你做非法事。真要使用务必小心！

# C 风格转换

- C++ 不鼓励使用这种显式转换，因此这里只略讲。
- `(Type)x` 或 `Type(x)`/`Type{x}` 都可以，但后两种有时像构造新对象或声明新函数，可能产生歧义。
- 我们不讲歧义如何消解，只用第一种写法。
- 顺便一提，C++23 打了个补丁：可用 `auto{…}`/`auto(…)` 得到表达式的 decay 后纯右值。标准措辞里常用来表示 decay copy。
- 其尝试顺序类似：
- 先 `const_cast`。
- 再 `static_cast` + `const_cast`（并可忽略访问控制，即 private 继承不影响 C 风格转换）。
- 再 `reinterpret_cast` + `const_cast`。

# 类型安全的 union 与 void*

- 我们已知 `union` 与 `void*` 相当危险。
- 由于生命周期，`union` 里只能使用活跃成员；使用另一成员是 UB。
- 同理，若把 `void*` 转为非 pointer-interconvertible 的类型，也是 UB。
- 天啊，UB、UB，为什么没有更明确的信号？
- 自 C++17 起引入 `<variant>` 与 `<any>` 来保证安全。
- `std::variant` 可视为带 `size_t` 下标的 `union`，在取值时会检查成员是否处于生命周期内。
- `std::any` 可视为带“魔法存储”的 `void*`，保存了原始类型，从而用错误类型取内部对象会失败。
- 它们的功能也更丰富。

# Union 的行为*

# any

- 可用它装载任意对象，例如：
- 更精确地说，`std::any` 要求底层对象有拷贝构造函数。
- 类型也会被 decay。
- 也可默认构造或调用 `.reset()`，此时不持有任何对象。

# any

- `std::any` 可以像 `std::function` 一样做 SBO（小缓冲区优化）。
- 正常来说 `sizeof(std::any)` 应是 `sizeof(void*)`。
- 但由于 SBO，GCC 是 16 字节，Clang 是 32 字节，MSVC 是 32/64 字节（取决于 x86/x64）！
- 即满足“noexcept 可移动构造”的小对象会直接放在栈上而不是堆分配，能略微降低时间开销。
- 最后一些辅助：
- `.swap` / `std::swap` / `.emplace`，与 `std::variant` 类似；
- `.type()`，类似底层对象的 `typeid`。
- `std::make_any()`，与构造 `std::any` 相同。
