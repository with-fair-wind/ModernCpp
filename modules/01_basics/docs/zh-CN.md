# auto

- 在谈这些类型之前，先介绍 C++11 的 `auto`！
- 凡是能从 `=` 右侧推导出的类型，都可以用 `auto` 代替书写！
- 例如 `auto a = 1;` ⇒ `auto` 为 `int`。
- `auto` 不会推导引用与 `const`，因此有时需要显式写出，例如 `const auto& a = b;`。
- 准确地说，这是类型退化（decay），后面还会讲到。
- 也可以显式写指针，例如 `auto* a = &b;`，不过 `auto a = &b;` 同样正确。
- 对“显然但复杂”的类型很有用，例如迭代器。
- 更多 `auto` 用法后面还会继续讲……

# 整数

- 那么整数有多大？
- 标准并未严格规定：
- `1 == sizeof(char) <= sizeof(short) <= sizeof(int) <= sizeof(long) <= sizeof(long long)`
- `1` 几乎总是 1 字节，但也可能是 4 字节等（某些 DSP 就是这样，令人头疼）。
- 以位宽计：`8 <= char`；`16 <= short, int`；`32 <= long`；`64 <= long long`
- 不过有一些广泛使用的数据模型：
- ILP32（4/4/4）：`int`、`long`、指针都是 4 字节。
- 广泛用于 32 位系统。
- LLP64（4/4/8）：`int`、`long` 为 4 字节；指针为 8 字节。
- 广泛用于 Windows。
- LP64（4/8/8）：`int` 为 4 字节；`long` 与指针为 8 字节。
- 广泛用于 Linux 与 macOS。`1 byte == 8 bits`（如今几乎是事实标准，但极少数架构可能无法单独按字节寻址，因此也可以说那里最小的可寻址单位是一字节）。

# 浮点数

- IEEE 754：符号位 + 阶码域 + 尾数域
- 尾数域 `0.x₁…xₙ = Σ … = M`
- Bias = `2^(k-1) - 1`，其中 `k` 为阶码长度。
- 非规格化数：阶码域为 0，值为 `M · 2^(exp - Bias)`（示意）
- 规格化数：值为 `(1 + M) · 2^…`
- 特殊值：阶码域全为 1
- 若尾数域为 0，则为 `inf`
- 否则为 `NaN`。
- 舍入模式：向偶数舍入/向 0/向下/向上等
- 对 `NaN`，除 `!=` 外的比较都为假；`&&`/`||` 中含 `NaN` 的情况；算术运算得到 `NaN`；`inf + (-inf) = NaN`
- `+inf` 大于所有非特殊值；`-inf` 最小。

# 引用类型

- 在 C++ 的某些场景里，引用可以看作指针的语法糖。
- 例如想在函数外修改变量，可以传指针。
- 引用省去了显式取地址，有时更易读。
- 区别：
- 引用不能为空。
- 引用不能改绑到别的对象。
- 引用可能不占内存，而指针通常会占；但编译器优化后这点往往不重要。
- 标准库的很多地方不允许引用作为模板实参，例如不能 `std::vector<A&>`。

# cv 限定符

- `const` 与 `volatile`。
- 在 ICS 的 ECF 中我们学过：`volatile` 用于强制编译器每次都从内存读取。
- 否则可能被优化为从寄存器读，并一直在寄存器上操作。
- 典型场景是异常处理程序修改了某些变量：普通函数不知道会被 OS 异常改掉，若你读两次，编译器可能优化成只从内存读一次、第二次从寄存器读。
- 还有其它“进程外改内存而程序不知道”的情况，也必须用 `volatile` 来刷新。
- `const`：把某物标记为不可修改。

# 数组类型

- 例如 `int a[3] = {1,2,3}, b[] = {4,5,6};`
- 大小可推导时可以省略 `3`。
- 若提供的元素少于给定大小，其余元素为 0。
- 但若一个元素都不提供，栈上数组的值是未初始化的随机值。
- 以前你可能把数组当成指针，但并不完全如此……
- 那其实是退化（decay）；类型安全一讲还会讲。
- 特殊的数组是 C 字符串，例如 `const char* str = "test";`
- `"test"` 的类型是 `const char[5]`，但会退化为 `const char*`。
- C++ 不允许用 `char*` 指向字符串字面量（丢掉 `const`）。
- 以空字符结尾，因此与普通 `char` 数组本身不同。
- 不能有“函数数组”或“引用数组”。
- VLA（即 `int arr[n]`）在 C++ 中不允许。

# 表达式

- 从编译器视角（编译原理课也会学），表达式其实是一棵树，由结合性与优先级决定。
- 优先级先用来切分各项，例如 `9/3/2+2*5` 会分成根 `+`：左子 `9/3/2`、右子 `2*5`，因为 `*`、`/` 优先级更高。
- 结合性决定树如何“生长”。
- 左结合会让树往左长，右结合往右长。
- 例如 `/` 左结合，所以 `9/3/2` 实际是根 `/`：左子 `9/3`、右子 `2`。
- 又例如 `/=` 右结合，所以 `a /= b /= c` 实际是 `a /= (b /= c)`；若 `a=4,b=2,c=2`，结果为 4。
- 每个表达式最终都会构造成一棵树（未必是二叉树，例如 `?:` 是三元，`++` 是一元）。

# 表达式

- 然后，通过表达式求值顺序来计算整棵树。
- 只保证：在求根节点之前，会先求左右子树；左右子树的相对顺序未指定。
- 例如 `f1() + f2() + f3()`，结构是根 `+`：左子 `f1()+f2()`，右子 `f3()`，而左子又是根 `+`：左 `f1()`、右 `f2()`。
- 我们知道在计算外层 `+` 之前，左右子树要先算完。
- 但可以先：
- 左子树先算 `f1()`
- 右子树先算 `f3()` 得到值
- 左子树再算 `f2()` 得到值
- 这仍符合规则：例如 `f1()` 与 `f2()` 都在左子树结果之前完成。
- 因此若在 `f1()`/`f2()`/`f3()` 里分别输出 `a`/`b`/`c`，`abc` 的任意排列都可能出现！
- 总之，求值顺序很大程度上取决于编译器如何计算这棵树。

# 表达式

- 此外，有些运算符有副作用，例如 `++`、`/=`。
- 于是更复杂：副作用何时发生？
- 例如臭名昭著的“谭式 C”：`++num + num`，其中 `num = 1`。
- 可以确定 `++num` 的值是 2，但若 `++` 的副作用在第二次求值 `num` 之前发生，则第二个 `num` 是 2；否则是 1。所以结果可能是 3 或 4！
- 这是未定义（UB）而不是未指定：同一编译器也可能表现不同。
- 有一些规则：
- 对 `&&` 与 `||`，由于短路，第一部分会完全求值，即值计算与副作用在第二部分之前完成。
- 对函数调用，所有参数（包括例如 `a.f()` 或 `a->f()` 里的 `a`）在进入函数前完全求值。
- 对 `condition ? exp1 : exp2`，`condition` 在 `exp1`、`exp2` 之前完全求值。
- 对逗号运算符 `,`，各子表达式按顺序完全求值。

# 表达式

- 自 C++17 起增加许多规则，使其更符合人类直觉。
- 函数参数按不确定顺序求值，但每个参数所代表的子树会以一种不重叠的方式完全求值！
- 这在某些场景很重要，后面会再讲。
- 每个重载运算符与内建运算符遵循相同的求值规则，而不是仅仅当作一次函数调用。
- 对 `E1[E2]`、`E1.E2`、`E1 << E2`、`E1 >> E2`，总是先完全求值 `E1` 再 `E2`。
- 对 `E1 = E2`、`E1 @= E2`，总是先完全求值 `E2` 再 `E1`。
- 在这些规则下，一些以前的怪异 UB 表达式变为合法，例如 `num = num++ + 2` 在 `num = 1` 时恒为 4。
- 以前可能是 2，因为 `num++` 的副作用可能发生在 `=` 之后，于是 UB。

# 表达式

- 但像 `n = ++num + num++` 仍是 UB。

- 总之，不建议把大量带副作用的表达式堆在一行里，无论从 UB 还是调试角度。
- 因为它们发生在“一步”里，bug 落在此处时很难单步进去查。
- 拆成多条表达式更友好。

- 表达式还有其它重要内容；会在移动语义一讲继续。

# 对象初始化

- 以前我们学过如何定义类的对象。
- 例如 `A a(1);` 或 `A a = 1;`。
- 自 C++11 起引入统一初始化（Uniform Initialization）。
-（几乎）所有初始化都能用花括号 `{}` 完成。
- 相比 `()`，它会严格检查窄化转换（Narrowing Conversion）；
- 即转换后的类型无法表示原类型的所有值。
- 例如用 `uint32_t` 初始化 `uint16_t` 会报错；若真要这么做需手动转换。
- 甚至 `uint32_t` 与 `int32_t` 互转也可能禁止，因为后者有负数前者无法全表示，前者有更大的正数后者无法全表示。
- 但若该值在编译期可确定且可表示，则仍合法（例如 `const uint32_t a = 2` ⇒ `uint16_t`）。
- 这有助于类型安全。
- 也能避免最令人头疼的解析问题：例如 `Class a();` / `Class a(C2());` 会被解析成函数声明而不是变量！

# 多态

- 调用纯虚函数看似被编译器禁止，那怎么做？
- 只有对象完全构造完成后，虚函数才按多态工作；否则 UB。
- 也可以直接认为：虚指针在构造函数体执行完毕后才按最终类型就绪。
- 因此，在派生类构造期间，虚指针实际指向基类虚表。
- 当你在普通成员函数里调用虚函数时，`this` 参与动态绑定，会按虚指针调用。
- 利用这一点，可以绕过编译器检查，因为编译器不知道普通函数最终会调用哪个虚函数！
- 于是若在构造函数里调用该普通函数，而它又去调纯虚函数，编译器找不到实现，运行期会出错（例如 MSVC 报 “pure virtual function call”）。

# 函数重载

- C++ 允许函数同名但参数不同。
- C 里禁止这样做。
- 编译器通过名字改编（name mangling）实现。
- 虽然两个函数“看起来同名”，编译器会把它们根据参数及类型装饰成不同符号。
- 返回类型不参与名字改编！
- 除非它是模板参数，因为所有模板参数都会进入改编名。
- 例如函数 `namespace Namespace {int function(int x);}`，MSVC 改编为 `?function@NameSpace@@YAHH@Z`，gcc 为 `_ZN9NameSpace8functionEi`。

# 运算符重载

- 一种特殊重载是运算符重载。
- 对某类的对象，运算符可通过重载获得不同含义。
- 运算符仍接受相同“位置”的操作数，不能发明新符号当运算符。
- 但操作数类型不必相同，例如可以写 `operator+(int, Vector3)`。
- 可以写成成员函数：
- 实际是 `a.operator+(b)`，因此对象必须是第一个操作数。
- 或写成全局函数：
- 实际是 `operator+(a, b)`，因此当 `b` 是 `Vector3` 时，`a` 可隐式转换为 `Vector3`。

# 运算符重载

- 可重载：
- `+,-,*,/,%,|,&,^,<<,>>`：推荐全局函数；返回新对象。
- 特别地，`<<` 与 `>>` 用作流插入/提取时必须是全局函数，因为你不能给流类加成员方法；后面会演示。

- `+,-,~,后缀++/--`：一元，推荐成员函数，返回新对象。
- 特别地，后缀 `++`/`--` 有一个未使用的 `int` 参数，用于区分前缀与后缀。

- `<=>` 实际返回 `std::strong_ordering`、`std::weak_ordering` 或 `std::partial_ordering`，定义在 `<compare>`。
- 对 `std::strong_ordering`，`a > b`、`a < b`、`a == b` 恰有一个为真，另两个为假。
- 例如整数比较。
- 对 `std::partial_ordering`，可能三者全假。
- 例如浮点比较：`NaN` 的 `>`/`<`/`==` 全假。
- 排序结果可与 `0` 比较。
- 与 `0` 的关系对应底层关系，例如 `<0` 表示小于。
- 也可用 `bool std::is_lt/eq/gt/lteq/gteq/neq(ordering)` 判断。
- 不能用于 `switch`，因为它不是整数。
- `std::weak_ordering` 没那么常用。
- 你可以想象：两件事并不严格相等，但仍让 `==` 为真，就属于这种弱序，例如忽略大小写的字符串。

# 运算符重载

- 对 `std::partial_ordering` 还有无序关系；当所有关系都为假时，即无序。
